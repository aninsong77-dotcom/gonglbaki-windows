# 보고서 작성 책상 — 요구사항 정의

> 작성일: 2026-07-08 · 작성: planner
> 관련 파일 조사 완료: `src/pages/CaseDrawer.tsx`, `src/pages/Index.tsx`, `server/app.py`, `~/.claude/skills/hwpx/scripts/*.py`

---

## 0. 문서 성격

이 문서는 두 단계로 만들어졌다.

1. 1차 대화에서 사용자가 확인해 준 12개 사실(보고서 여러 종류, hwpx 양식, `{{플레이스홀더}}` 방식, 온디바이스 원칙 등)
2. 사용자가 직접 준 핵심 설계 설명("양식이 구조를 정의하는 범용 폼 생성기") — 이것이 **이 기획의 중심 축**이며, 1차 대화에서 planner가 세웠던 "보고서 종류별 고정 구조" 전제를 대체한다.

남은 세부사항 중 사용자가 직접 정하지 않은 부분은, 이 문서 안에서 **"이렇게 설계하겠습니다 — 다르면 말씀해주세요"** 형태의 기본안으로 명시한다. 별도 질문 뭉치로 되돌리지 않는다.

## 1. 목적

- 사례서랍에 연결된 자료(축어록·심리검사·가계도·회기요약 등)를 참고하면서, 사용자가 미리 만들어 둔 hwpx 양식에 내용을 채워 넣어 보고서를 완성하는 화면을 추가한다.
- 목표는 두 가지: **작성 시간 절약**(반복 서식·표 작업 자동화, AI 초안 제공) + **결과물의 정형성**(양식 그대로, 사람이 편집기와 씨름할 필요 없음).
- 온디바이스 원칙 유지: 축어록·사례 데이터·AI 처리 전부 이 PC 안에서만.

---

## 2. 핵심 설계 원칙 (사용자 확정)

**보고서 "종류"를 앱이 미리 알 필요가 없다.** 참관자 보고서·발표자 보고서·심리평가 보고서를 앱에 하드코딩하지 않는다. 대신:

- 사용자가 hwpx 양식 파일을 앱에 **등록**한다 (양식 1개 = 보고서 "종류" 1개, 이름은 사용자가 붙임).
- 양식 파일 안에는 섹션 경계를 나타내는 마커와 필드 플레이스홀더가 미리 심어져 있다 (§4 참조).
- 앱은 그 마커를 읽어 **"제목 + 내용 입력칸"을 자동으로 생성**한다. 참관자 보고서든 심리평가 보고서든 전부 같은 메커니즘으로 동작 — 양식만 다르게 등록하면 새 보고서 종류가 생긴다.
- 모든 칸을 채우면, 프로그램이 각 칸의 내용을 hwpx 양식의 정확한 위치에 넣어 보고서를 완성한다.

## 3. 화면 흐름

```
사례서랍 (CaseDrawer)
  └─ 사례 카드 안 "보고서 작성 책상" 버튼 (신규)
       └─ 보고서 종류 선택 (등록된 양식 목록에서 고르기, 없으면 "양식 등록"으로 유도)
            └─ 보고서 작성 화면 (좌우 이등분)
                 좌: 참고자료 패널 (탭 전환)
                 우: 작성 패널 (양식에서 뽑은 섹션들, 세로 나열)
                      └─ "보고서 완성" → hwpx 생성 → 사례 폴더에 저장
```

### 3.1 참고자료 패널 (좌측)

- 그 사례에 연결된 자료를 **탭**으로 전환하며 열람 (읽기 전용).
- 탭 후보: 회차별 축어록(json/word) · 회차별 분석(기존 AI 요약 결과, 있으면) · 가계도 · 심리검사 보고서(복수) · 슈퍼비전 자료(복수).
- 미리보기는 기존 `FilePreviewDialog`의 로직을 재사용(오디오 재생·이미지·텍스트 표시, 그 외 형식은 "시스템 기본 프로그램으로 열기" 폴백) — 모달이 아니라 패널 내장 형태로 이식.
- 회차가 여러 개면 회차 선택 드롭다운 → 그 회차의 자료가 탭에 반영.

### 3.2 작성 패널 (우측)

양식에서 추출한 섹션을 순서대로 세로 나열. 섹션은 3가지 유형을 가진다 (§4.2).

| 섹션 유형 | UI |
|---|---|
| 일반 텍스트 | 제목 + 여러 줄 textarea. 내담자 기본정보처럼 이미 앱에 있는 데이터는 자동으로 채워짐(수정 가능). |
| AI 초안 섹션(회기요약형) | 제목 + textarea + "AI 초안 생성" 버튼. 좌측 패널에서 현재 선택된 회기를 기준으로 초안 생성, 사람이 검토·수정. |
| 표/붙여넣기 섹션 | 제목 + "표를 여기에 붙여넣으세요" 전용 붙여넣기 칸. 클립보드에서 표를 그대로 붙여넣으면 표 형태로 인식되어 미리보기 표시. |
| 길잡이(가이드) 섹션 | 제목 + 하위 질문 여러 개(예: 1.내방계기 2.주호소문제 ...)를 순서대로 입력 → 저장 시 자동으로 하나의 문단으로 연결. |

하단 "보고서 완성" 버튼 — 모든 필수 섹션이 채워지면 활성화, hwpx 생성 후 저장 위치 안내.

## 4. hwpx 양식 등록 · 섹션 인식 방식 (기술 판단 — 기본안)

### 4.1 기존 hwpx 스킬 능력 확인 결과

`~/.claude/skills/hwpx/scripts/`를 직접 읽어 확인했다.

- `hwpx2md.py`: hwpx 문단·표를 markdown으로 변환(텍스트 추출용). **문단의 스타일(제목/본문 구분)은 추출하지 않음.**
- `extract.py` / `template.py`: 양식 hwpx에서 대표 서식(자간·장평·줄간격·폰트)을 뽑아 "프로파일"을 만드는 도구. **"가장 크고 굵은 글씨 = 제목"이라는 휴리스틱**을 쓰지만, 이건 md→hwpx **신규 생성**시 제목 스타일을 정하기 위한 것이지, **기존 hwpx 안에서 "여기부터 여기까지가 한 섹션이다"를 안정적으로 식별하는 용도가 아니다.**
- `build.py`: markdown의 `| ... |` 표 문법을 hwpx 표(`hp:tbl`)로 변환하는 기능이 이미 있음 — **표 삽입 자체는 재사용 가능한 기반 기술이 있다.**
- 결론: **"제목 글씨 크기로 섹션 자동 인식"은 신뢰도가 낮아 채택하지 않는다.** 실사용 양식마다 제목 스타일이 다르게 잡혀 있을 수 있어 오탐 위험이 큼.

### 4.2 기본안: 마커 기반 섹션 인식 (플레이스홀더 방식의 연장)

사용자가 이미 합의한 `{{내담자명}}` 방식을 섹션 단위로 확장한다.

```
{{SECTION:내담자 이해의 확장}}
{{/SECTION}}

{{TABLE:심리검사 결과}}
{{/TABLE}}

{{GUIDE:사례개념화}}
1. 내방계기
2. 주호소 문제
3. 가족력·생육사
4. 문제의 이해(심리적 기제)
5. 개입 방향
{{/GUIDE}}
```

- 앱이 hwpx를 열어 이 마커 쌍을 찾아 순서대로 섹션 리스트를 만든다. 마커 사이의 원래 내용(있다면)은 안내문으로 취급하고, 실제 채움은 마커 자리를 치환한다.
- `{{GUIDE:...}}` 안의 번호 목록은 **하위 질문 프리셋**으로 파싱된다. 프리셋은 양식마다 다르게 사용자가 직접 정의(하드코딩 아님) — 별도 설정 화면 불필요, 양식 파일 자체가 설정이 된다.
- 이 방식을 고른 이유: (a) 파싱이 결정적(문자열 매칭)이라 오탐이 없음 (b) 사용자가 이미 플레이스홀더 방식에 익숙해지기로 확인한 것과 동일한 사고방식이라 새 학습부담이 적음 (c) 스타일 휴리스틱보다 구현·검증이 훨씬 단순해 이번 트랙의 리스크를 낮춘다.
- **안전망**: 양식 등록 시 자동 인식된 섹션 목록을 확인 화면에서 보여주고, 순서·제목 표시명을 사용자가 조정할 수 있게 한다(마커 자체를 잘못 심었을 경우 대비).

> 다르면 말씀해주세요 — 예를 들어 이미 갖고 계신 실제 양식 파일에 `{{SECTION:...}}` 마커를 심는 게 번거로우시면, "일반 텍스트 필드는 종전처럼 `{{필드명}}` 단일 치환, 섹션 구분은 필요한 곳에만 마커를 추가" 식으로 최소 침습으로도 설계할 수 있습니다.

### 4.3 표 붙여넣기 처리 (기본안)

- 붙여넣기 칸은 브라우저의 클립보드 `paste` 이벤트에서 HTML 표(`<table>`)를 받아 파싱 → 내부적으로 `행×열` 구조로 저장, 화면에는 그리드로 미리보기.
- hwpx 저장 시 이 구조를 `build.py`의 markdown 표 변환 로직과 동일한 방식으로 `hp:tbl` XML을 만들어 `{{TABLE:...}}` 마커 위치에 삽입.
- 사용자가 말한 "심리평가 자료의 특정 부분을 잘라서 붙여넣는" 시나리오는, 왼쪽 참고자료 패널에서 심리검사 보고서 파일을 열람하며 그 안의 표를 직접 선택·복사해 오른쪽 표 칸에 붙여넣는 흐름으로 충족한다. **표 내용을 앱이 자동으로 추출·인식하는 기능(자동 파싱)은 이번 범위에 포함하지 않는다** — 붙여넣기는 사람이 수행, 앱은 "붙여넣은 걸 정확한 위치에 정확한 표 형태로 꽂아주는" 역할까지만.

## 5. AI 초안(회기요약 재구성) 설계 방향

### 5.1 기존 기능과의 관계

`server/app.py` 596~900행에 이미 온디바이스 회기요약 파이프라인이 있다: `Qwen3-4B-Q4_K_M.gguf` 모델을 `llama-completion.exe`로 구동, 축어록을 조각내 map(부분 요약)한 뒤 reduce(종합) — 화자 라벨이 있으면 `[내담자]/[상담자]` 구조로 나누고, `all`/`nae`/`sang` 세 scope로 재종합 가능. `src/pages/Index.tsx`의 "분석" 탭에서 사용되며, **결과는 화면에 표시·수정 가능한 상태로만 존재하고 CaseDrawer의 `analysis` 슬롯 파일과 자동 연동되지 않는다** (사용자가 "결과 저장"을 눌러야 텍스트 파일로 남음).

### 5.2 이번 기능에서의 재사용 방식 (기본안)

- 기존 map-reduce 인프라(모델·실행 방식·SSE 스트리밍)는 그대로 재사용한다. 다만 reduce 단계의 프롬프트를 새로 짠다.
- 새 프롬프트는 **① 기존 AI 회기요약(이미 만들어져 있다면) + ② 축어록 원문 조각 요약**을 함께 입력으로 받아, "보고서에 바로 쓸 수 있는 문어체 초안"을 만들도록 지시한다(기존 프롬프트는 대화체 요약에 가까움 — 보고서용은 항목화·격식체가 필요).
- 워크플로우: 사람이 "AI 초안 생성" 버튼을 누름 → 좌측 패널에서 **현재 선택된 회기**의 축어록을 기준으로 초안 생성 → 우측 textarea에 채워짐(사람이 그대로 쓰거나, 일부만 쓰거나, 다시 씀) — **AI가 완전자동으로 최종본을 만들지 않는다.**
- 회기 범위 기본안: 여러 회기가 있는 사례에서는 **"지금 좌측 패널에서 열람/선택 중인 회기"를 기준으로 초안 생성**을 기본으로 한다. 발표자 보고서처럼 특정 1회기가 필요한 경우와, 심리평가 보고서처럼 전체 기간 종합이 필요한 경우가 섞여 있을 수 있으므로, AI 초안 생성 버튼 옆에 "이 회기만" / "선택한 여러 회기 종합"을 고를 수 있는 간단한 토글을 둔다(전체 자동 종합 여부를 섹션마다 사용자가 그때그때 정함 — 보고서 종류별로 앱이 미리 정해두지 않음. §2 원칙과 일관).

> 다르면 말씀해주세요 — "이 회기만 / 선택한 여러 회기 종합" 토글이 과하다면, 우선 "이 회기만" 단일 동작으로 좁혀서 먼저 만들 수도 있습니다.

## 7. 저장 위치 (기본안)

- 완성된 보고서는 **사례 폴더 안에 저장**한다. 기존 사례서랍 파일 저장 패턴(경로로 연결, 실체는 파일시스템)과 일관성을 위해서다.
- 기본 경로: `<사례폴더>/보고서/<보고서명>_<날짜>.hwpx`
- `CaseRecord`에 `reports: ExtraDoc[]` 필드를 신설해, 완성된 보고서를 기존 `psychTests`/`supervisions`처럼 사례서랍 안에서 목록으로 보고 다시 열 수 있게 한다.
- "다른 위치로 내보내기"는 완성 직후 안내 토스트에 "파일 위치 열기" 버튼 정도로 보조 제공(별도 export 플로우를 새로 만들지 않음 — 이미 파일이 사례 폴더에 있으니 탐색기에서 복사하면 됨).

> 다르면 말씀해주세요 — 사례 폴더 밖 임의 위치로 내보내기가 자주 필요하시면(예: 인쇄소·타 기관 제출용 별도 폴더), "완성 시 저장 위치 선택 다이얼로그"를 추가하는 것도 어렵지 않습니다.

## 8. 아키텍처 판단 (hwpx 조작 위치) — 기본안

**Python Flask 백엔드(`server/app.py`)에서 처리한다.** 근거:

- hwpx 읽기·서식추출·표 생성 로직이 이미 Python(`~/.claude/skills/hwpx/scripts/`)으로 존재 — 이걸 이식/재사용하면 Node.js로 zip+XML 파싱을 새로 짜는 것보다 리스크가 훨씬 낮다.
- `server/app.py`는 이미 파일 시스템·서브프로세스(whisper, llama.cpp)를 다루는 무거운 작업을 전담하는 계층이며, Electron 메인 프로세스(`electron-main.cjs`)는 지금까지 창 제어·서버 spawn·IPC(`preload.cjs`)만 담당해 왔다 — 기존 레이어 책임 분리와 맞다.
- Electron 메인/`preload.cjs`는 기존 `caseDrawerAPI`처럼 파일 선택 다이얼로그(양식 파일 선택 등) IPC만 담당, 실제 hwpx 파싱·생성은 프론트가 새 Flask 엔드포인트(`/api/report/...`)를 호출.

상세 API·모듈 구조는 `structure.md`에 정리.

## 9. 제외 범위 (이번 기획에 안 넣는 것)

- 보고서 종류·양식의 사전 정의(참관자/발표자/심리평가 등 하드코딩) — 대신 사용자가 양식을 등록하는 범용 구조로 대체.
- 표 내용의 자동 추출/자동 인식(예: 심리검사 PDF·이미지에서 점수표를 자동으로 읽어오는 OCR류 기능) — 붙여넣기는 사람이 수행.
- hwpx 양식 자체를 앱 안에서 새로 디자인/제작하는 기능(마커·필드 배치는 사용자가 한글 프로그램에서 직접 준비).
- 여러 보고서를 한 번에 일괄 생성(batch)하는 기능.
- 보고서 버전 관리·이력 비교(같은 보고서를 여러 번 고쳐 쓴 이력 추적).
- 사례 폴더 밖으로의 자동 내보내기·인쇄·PDF 변환 파이프라인.
- 협업/공유(여러 사람이 같은 보고서를 동시 편집) — 이 앱은 로컬 단일 사용자 전제 유지.

